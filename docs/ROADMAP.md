# tssp — Engineering Specification

**Project:** `tssp` — self-hosted transfer & storage system for Orange Pi
**Spec version:** 1.0
**Audience:** Implementation engineer or AI coding agent (Claude Code / Codex)
**Status:** Ready for implementation. This document is authoritative.

---

## 0. How to Read This Document

This specification defines **what must be built and how it must behave**. It deliberately does not prescribe:

- The on-disk project layout (folder and file names within the source tree).
- The internal file storage hierarchy on the server (the implementer chooses what is safe and efficient).
- Concrete code snippets, function signatures, or type definitions.

It does prescribe, with no room for interpretation:

- The technology stack to use.
- Every CLI command, flag, and observable behavior.
- Every HTTP endpoint and its semantics.
- Every edge case that must be handled and how.
- The quality bar the codebase must meet.

Where this document uses **MUST**, **MUST NOT**, **SHOULD**, **SHOULD NOT**, and **MAY**, those words carry RFC 2119 meaning. "MUST" is non-negotiable.

---

## 1. Mission

Build a **production-grade, self-hosted file transfer and storage system** that runs on a passively cooled Orange Pi and replaces ad-hoc file sharing (WhatsApp, email, USB sticks, browser uploads). The system must be:

- **Fast.** Sub-second response for metadata queries. Line-rate uploads and downloads on local Wi-Fi.
- **Light.** Idle RAM footprint under 30 MB for the backend daemon. Negligible CPU when idle. No background polling loops.
- **Reliable.** Power-loss safe. Never corrupts the index. Never loses a file that was acknowledged as uploaded.
- **Scriptable.** First-class CLI with stable output formats and exit codes.
- **Beautiful.** CLI output that respects modern terminal conventions (colors, progress, Unicode) and degrades cleanly when those aren't available.
- **Maintainable.** A new contributor can read the codebase, locate any feature, and ship a change in under an hour.

Quality bar: **comparable to `ripgrep`, `sqlite`, `restic`, or the `curl` codebase.** This is the standard against which all decisions are measured.

---

## 2. Deliverables

Three artifacts ship together under one repository and one version number:

1. **`tsspd`** — the backend daemon. A single self-contained binary that runs on the Orange Pi.
2. **`tssp`** — the CLI client. A single self-contained binary that runs on any user machine (Linux, macOS, Windows).
3. **`tssp-web`** — the embedded web dashboard. A static bundle served by `tsspd` at the root path. Mobile-first progressive web app. Detailed UX is **out of scope for this spec** and will be designed separately; the backend must expose every capability the web app will eventually need.

Additionally:

4. **Shell completion scripts** for `bash`, `zsh`, `fish`, and PowerShell. Generated by the CLI itself via a hidden `tssp completions <shell>` subcommand.
5. **Manual pages** for `tssp` and `tsspd`, generated from the same source of truth as the CLI definitions.
6. **A reference HTTP API document** generated from the server's route definitions, not hand-written.

## 3. Non-Negotiable Quality Principles

These override convenience, delivery speed, and personal preference. A pull request that violates any of these MUST be rejected.

1. **Zero panics in production paths.** Every `Result` and `Option` is handled. `unwrap`, `expect`, and indexing operations that can panic are forbidden outside of test code and one-time startup invariants that are documented inline with a comment explaining why the invariant holds.

2. **Streaming everywhere.** No file is ever fully loaded into memory. Uploads, downloads, hashing, thumbnail generation, and any transformation must stream. The system must handle a 10 GB file on a device with 1 GB of RAM without swap pressure.

3. **Hexagonal architecture.** The core domain logic MUST NOT depend on HTTP, on the filesystem, on SQLite, on the clock, or on the network. Those are adapters. The core must be unit-testable in-memory with no I/O.

4. **Modular crates.** The repository is a Cargo workspace. Each subsystem is its own crate with a minimal, documented public API. The dependency graph between crates is acyclic and reviewed at every change.

5. **Typed errors at boundaries.** Library code uses `thiserror` enums with structured fields. `anyhow` is permitted only at the outermost binary boundary (the `main` function of `tssp` and `tsspd`) and in tests.

6. **No global mutable state.** No `static mut`, no `lazy_static` for mutable data, no `OnceCell` holding application state. State is owned by an application context type that is passed explicitly.

7. **Fail loudly, recover silently.** All errors are logged with structured context (operation, inputs, error chain). User-facing messages are short, actionable, and never expose internals like file paths on the server or stack traces.

8. **Backward-compatible interfaces.** Once shipped, the CLI flag surface and the HTTP API are contracts. Breaking changes require a major version bump and a documented migration. Adding new flags, endpoints, or response fields is always allowed and is non-breaking.

9. **Reproducible builds.** `Cargo.lock` is committed. The Rust toolchain version is pinned via `rust-toolchain.toml`. CI builds must be byte-identical across runs on the same platform.

10. **Observable by default.** Every request, every file operation, every error emits a structured log event. Critical paths increment counters. A user reporting a bug attaches a log and the maintainer can reconstruct what happened end-to-end.

11. **Concurrency-safe.** Two clients uploading the same file at the same moment, two clients tagging the same file, a client downloading a file while another is deleting it — all of these must produce a defined, correct outcome. There are no races, no torn writes, no orphaned data.

12. **Idempotent where it counts.** Uploading the same content twice produces one stored file and a clear signal to the client that this was a deduplication. Adding the same tag twice is a no-op, not an error. Deleting an already-deleted file returns success with a note.

## 4. Technology Stack — Required

The implementer MUST use the following. These are not suggestions.

### Backend daemon (`tsspd`)

- **Language:** Rust, edition 2021 or newer, stable channel only.
- **Async runtime:** Tokio, multi-threaded scheduler, with the worker thread count configurable and defaulting to the number of physical CPU cores capped at 2 on ARM single-board computers.
- **HTTP framework:** `axum`. Justification: minimal overhead, first-class streaming via `Body`, ergonomic extractors, mature middleware ecosystem (`tower`, `tower-http`).
- **HTTP server:** `hyper` (used internally by `axum`). HTTP/1.1 mandatory. HTTP/2 cleartext support optional but SHOULD be enabled.
- **Metadata store:** SQLite via `sqlx` with the `sqlite` feature and offline query checking, or `rusqlite` with the `bundled` feature. WAL journal mode MUST be enabled. Synchronous mode set to `NORMAL`. Full-text search via the FTS5 extension. The SQLite library is statically linked — no system dependency.
- **Migrations:** versioned, forward-only, embedded in the binary. The daemon runs pending migrations on startup and refuses to start if migration fails.
- **Serialization:** `serde` with `serde_json` for API payloads, `toml` for configuration files.
- **Logging:** `tracing` with `tracing-subscriber`. Two output formats: human-readable pretty-printed for development, single-line JSON for production. Log level configurable per-module via `RUST_LOG`.
- **Error handling:** `thiserror` throughout library code, `anyhow` only in the binary entry point.
- **Configuration loading:** `figment` or equivalent. Layered: built-in defaults → config file → environment variables prefixed `TSSPD_` → CLI flags. Each layer overrides the previous.
- **File hashing:** `blake3`. Justification: faster than SHA-256 on ARM, cryptographically strong, supports parallel and streaming hashing.
- **Image processing:** the `image` crate, used only for thumbnail generation, behind a Cargo feature flag so it can be disabled at build time.
- **QR generation:** the `qrcode` crate. Render to terminal (Unicode half-blocks), to PNG, and to SVG.
- **ID generation:** UUIDv7 (time-ordered) via the `uuid` crate. Justification: sortable by creation time, no central coordinator needed, opaque to clients.
- **Time:** `time` or `chrono`. All timestamps stored as UTC. Conversion to local time only at presentation.
- **Filesystem operations:** `tokio::fs` for async paths, `std::fs` only where necessary. Atomic writes via write-to-tempfile-then-rename on the same filesystem.

### CLI client (`tssp`)

- **Language:** Rust, same toolchain as the daemon, sharing a workspace.
- **Argument parsing:** `clap` v4 with derive macros. Subcommands and flags MUST be defined declaratively so manual pages and shell completions can be generated from the same source.
- **Progress bars:** `indicatif`. Bars MUST be drawn only when stdout is a TTY. When stdout is piped or redirected, progress information is written to stderr as plain newline-delimited status lines.
- **Terminal styling:** `owo-colors` or `nu-ansi-term`. MUST respect `NO_COLOR`, `CLICOLOR`, `CLICOLOR_FORCE`, and the absence of a TTY. MUST detect terminal capabilities (8-color, 256-color, true-color) and degrade.
- **HTTP client:** `reqwest` with `rustls-tls` (no OpenSSL). Connection pooling enabled. Configurable timeouts.
- **Clipboard:** `arboard`. MUST handle X11, Wayland, macOS, and Windows transparently. MUST degrade gracefully (informative error, non-zero exit) on headless systems.
- **Configuration loading:** same layered model as the daemon, with `TSSP_` environment prefix.
- **Config file location:** platform-appropriate via the `directories` or `etcetera` crate. Linux: `$XDG_CONFIG_HOME/tssp/config.toml`. macOS: `~/Library/Application Support/tssp/config.toml`. Windows: `%APPDATA%\tssp\config.toml`.
- **Shell completion generation:** via `clap_complete`. The CLI exposes a hidden `tssp completions <shell>` subcommand that prints the completion script to stdout.
- **Man page generation:** via `clap_mangen`, run at build time by `build.rs` and installed alongside the binary.

### Web dashboard (`tssp-web`)

- **Framework:** SvelteKit in static-adapter mode, producing a fully prerendered SPA bundle. Justification: smallest runtime payload, simplest mental model, best fit for a low-power server.
- **Bundling:** Vite, default configuration with production optimizations.
- **Styling:** Tailwind CSS or vanilla CSS with custom properties. No CSS-in-JS runtime.
- **Embedding:** the built static bundle is embedded into the `tsspd` binary via `rust-embed` or `include_dir`. No separate web server, no separate deployment.
- **PWA:** service worker for offline shell, web manifest for installability. Cache strategy is documented in the web app's own spec (separate document).

## 5. Architectural Model

### 5.1 Layering

The codebase is organized as a Cargo workspace with the following logical layers. The implementer chooses crate names and directory names; the layering itself is mandatory.

1. **Domain core** — pure logic, no I/O. Defines entities (file record, tag, pin, QR session), value objects (hash, file size, MIME type), and the rules that govern them (validation, state transitions). Depends on nothing.

2. **Ports** — trait definitions for everything the core needs from the outside world: persistence, blob storage, clock, ID generator, hasher, thumbnail engine. Depends only on the domain core.

3. **Adapters** — concrete implementations of the ports. SQLite adapter for persistence, filesystem adapter for blob storage, system clock adapter, UUIDv7 adapter, BLAKE3 adapter, `image` crate adapter. Each adapter is its own crate.

4. **Application services** — orchestrate the core and the ports to execute use cases (upload a file, list recent files, generate a QR session, etc.). Transactional boundaries live here.

5. **Delivery layer** — the HTTP server and its handlers. Translates HTTP requests into application service calls and translates application results back into HTTP responses. No business logic.

6. **Binary** — the `tsspd` executable. Wires everything together, loads configuration, starts the runtime, installs signal handlers, runs migrations.

The CLI follows the same model on its side: domain → ports → adapters → application services → command handlers → binary. The HTTP client is one adapter behind a `Backend` port; the clipboard is another adapter behind a `Clipboard` port. This means the CLI is unit-testable end-to-end against an in-memory fake backend.

### 5.2 Inter-process contract

`tssp` ↔ `tsspd` communicate only via the HTTP API. There is no shared library, no shared on-disk format, no shared IPC. The CLI MUST work against any future reimplementation of the daemon as long as the HTTP contract is honored, and vice versa.

### 5.3 Extensibility hooks

The architecture MUST anticipate, without implementing, the following future capabilities. Each one should be expressible by adding a new adapter or a new application service without modifying existing modules:

- Alternative blob backends (S3-compatible object storage, encrypted local volume).
- Alternative metadata stores (PostgreSQL for multi-user deployments).
- Auto-tagging via embedding models.
- Webhook delivery on file events.
- WhatsApp bot integration.
- Multi-user authentication and per-user quotas.
- End-to-end encryption.
- Federation between multiple Pi instances.

The implementer MUST NOT add code for these, but MUST design ports and event types in a way that does not preclude them.

## 6. Backend Daemon — `tsspd`

### 6.1 Responsibilities

The daemon is the single source of truth. It:

- Accepts file uploads and stores them durably.
- Serves file downloads as streamed responses.
- Maintains a metadata index that supports listing, searching, filtering, and ordering.
- Manages tags and pins.
- Issues short-lived QR-based transfer sessions.
- Generates and serves image thumbnails on demand or eagerly, with caching.
- Reports its health, storage usage, and operational metrics.
- Serves the embedded web dashboard.

### 6.2 Subsystems

Each subsystem is a distinct module with a defined responsibility, a defined public interface, and isolated tests.

**Storage subsystem.** Owns the on-disk layout of stored files. Decides where bytes go, how files are grouped, how naming collisions are avoided, how partial writes are made atomic, and how orphaned files are detected and reclaimed. The rest of the system addresses stored files only by opaque storage handles, never by path. The implementer chooses the on-disk layout; the only requirements are: (a) two files with identical content MUST share storage (content-addressed by BLAKE3 hash); (b) a power loss at any moment MUST leave the storage layer in a consistent state, with no half-written files visible to the index; (c) the layout MUST scale to at least one million files without filesystem performance degradation (commodity ext4 starts to slow at ~10,000 entries per directory — design accordingly).

**Index subsystem.** Owns the SQLite database. Exposes operations for inserting file records, updating tags and pins, querying by id / name / tag / date / pin status, and full-text searching filenames. All writes go through prepared statements. All reads return typed records. Schema changes go through versioned migrations.

**Upload subsystem.** Coordinates the storage and index subsystems to atomically ingest an uploaded file: stream the body to a temporary location, hash it on the fly, check whether the content already exists, deduplicate if so, otherwise move the bytes into permanent storage, then insert the index record. If any step fails, the partial state is fully cleaned up. The upload subsystem MUST NOT return success until both the file bytes and the index record are durable (fsynced).

**Download subsystem.** Looks up a file by id (or by other addressing modes, see API), opens the underlying blob, and streams it to the response. MUST support HTTP range requests for resumability and partial downloads. MUST set correct `Content-Type`, `Content-Length`, `Content-Disposition`, `ETag`, and `Last-Modified` headers. MUST handle the case where the index record exists but the blob is missing (return 410 Gone, log a corruption alert).

**Tag subsystem.** Manages the many-to-many relationship between files and tags. Tags are case-insensitive on lookup, stored as the user originally typed them. Tag names are normalized (trimmed, collapsed whitespace), validated against a defined character set, and length-limited. Listing all tags returns counts.

**Pin subsystem.** A pin is a boolean flag on a file record plus an ordering. The user can reorder pinned files. Operations are listing, pinning, unpinning, and reordering.

**Session subsystem (QR transfers).** Manages short-lived, single-use, unguessable transfer tokens. There are two session types: **send sessions** (a file already on the server, made available for one-time anonymous download) and **receive sessions** (a placeholder waiting for one anonymous upload). Tokens are 128 bits of entropy, encoded as URL-safe base64. Default TTL is 10 minutes for send sessions and 5 minutes for receive sessions, both configurable. Expired sessions are reaped by a background task that runs every 60 seconds. A session MUST be invalidated immediately upon successful use.

**Thumbnail subsystem.** Generates thumbnails for image files. Triggered on upload (eager) or on first request (lazy) — the choice is configurable, default lazy. Thumbnails are themselves stored via the storage subsystem and indexed by the source file's hash plus a size parameter. Thumbnails MUST never be generated synchronously on the upload request path if eager mode is enabled; that work happens on a separate worker queue.

**Health subsystem.** Exposes liveness (is the process running), readiness (can it accept requests), and detailed status (storage usage, index size, file counts, uptime, version). Used by the CLI's `status` command and by external monitors.

**Configuration subsystem.** Loads, validates, and exposes the daemon's configuration. Validation runs at startup and the daemon refuses to start with invalid configuration, printing a clear error.

**Lifecycle subsystem.** Owns startup, graceful shutdown, signal handling, and the orderly draining of in-flight requests. SIGTERM and SIGINT trigger graceful shutdown with a configurable timeout (default 30 seconds), after which in-flight requests are cancelled.

### 6.3 HTTP API Surface

All endpoints return JSON unless they stream a binary payload. All responses include a `Content-Type`. All error responses have the shape `{ "error": { "code": "<machine_readable>", "message": "<human_readable>", "details": { ... } } }`. HTTP status codes are used correctly (200/201/204 for success, 400 for client error, 404 for missing, 409 for conflict, 410 for gone, 413 for too large, 416 for bad range, 500 for server error, 503 for unavailable).

All endpoints accept and return `application/json` for bodies unless otherwise specified. All endpoints support `OPTIONS` preflight for CORS.

The endpoint surface, grouped by resource:

**Files:**
- `POST /api/v1/files` — upload one file. Multipart/form-data with fields: the file part, optional tags (repeated), optional initial pin flag, optional destination hint. Returns the created file record with status 201. Idempotent on content hash: re-uploading identical content returns the existing record with status 200 and a header indicating deduplication.
- `POST /api/v1/files/batch` — upload many files in a single request. Multipart with multiple file parts. Returns a result object listing each file and its outcome (created, deduplicated, or failed with reason). Partial failure is allowed; the overall response is 200 with per-file status.
- `GET /api/v1/files` — list files. Query parameters for filtering (tag, since, until, mime prefix, name substring, pinned-only), ordering (uploaded-at, name, size), and pagination (cursor-based, not offset-based). Default ordering is uploaded-at descending. Default page size is 50, maximum 500.
- `GET /api/v1/files/{id}` — get a single file's metadata.
- `GET /api/v1/files/{id}/content` — download the file. Supports `Range`. Sets `Content-Disposition: attachment; filename=...` by default; supports `?disposition=inline` for in-browser preview.
- `GET /api/v1/files/{id}/thumbnail?size={small|medium|large}` — get a thumbnail. Returns 404 if the file is not an image and 202 with a retry hint if the thumbnail is being generated.
- `DELETE /api/v1/files/{id}` — delete a file. Returns 204. Idempotent: deleting a non-existent file returns 204 with a header indicating the file was already gone.
- `PATCH /api/v1/files/{id}` — partial update for renaming or moving (within the logical namespace exposed to clients, not the on-disk layout).

**Tags:**
- `GET /api/v1/tags` — list all tags with file counts.
- `POST /api/v1/files/{id}/tags` — add tags. Body is a list of tag strings. Idempotent.
- `DELETE /api/v1/files/{id}/tags/{tag}` — remove a tag. Idempotent.

**Pins:**
- `GET /api/v1/pins` — list pinned files in pin order.
- `PUT /api/v1/files/{id}/pin` — pin a file (optionally at a specific position).
- `DELETE /api/v1/files/{id}/pin` — unpin.
- `POST /api/v1/pins/reorder` — reorder pins.

**Search:**
- `GET /api/v1/search?q={query}` — full-text search across filenames and tags. Returns ranked results.

**Sessions (QR):**
- `POST /api/v1/sessions/send` — create a send session for an existing file. Body identifies the file; response includes the token, the download URL, and the expiration.
- `POST /api/v1/sessions/receive` — create a receive session. Response includes the token, the upload URL, and the expiration.
- `GET /api/v1/sessions/{token}` — get session status (also used by the CLI to poll for completion of a `receive` session).
- `GET /s/{token}` — public, unauthenticated download endpoint used by the receiving phone. Streams the file. Invalidates the session on successful response.
- `POST /u/{token}` — public, unauthenticated upload endpoint used by the receiving phone. Accepts multipart. Invalidates the session on successful response.

**System:**
- `GET /api/v1/status` — overall status: version, uptime, storage usage, file count, recent activity summary.
- `GET /healthz` — liveness probe, returns 200 always when the process is up.
- `GET /readyz` — readiness probe, returns 200 only when the daemon can serve requests.
- `GET /metrics` — Prometheus-format metrics (optional, behind a config flag).

**Web app:**
- `GET /` and `GET /<any-non-api-path>` — serves the embedded web dashboard (with SPA fallback routing).

### 6.4 Versioning

The API path is versioned (`/api/v1/`). A breaking change requires `/api/v2/`. Both versions can coexist during a transition period. The CLI sends an `Accept: application/vnd.tssp.v1+json` header so the daemon can also negotiate via content type if needed in the future.

### 6.5 Lifecycle

Startup sequence:
1. Load and validate configuration.
2. Initialize logging.
3. Open and migrate the database.
4. Verify the storage directory is writable.
5. Reap any stale session and temp-upload state from the previous run.
6. Start the HTTP listener.
7. Start background workers (thumbnail generator, session reaper, integrity scanner).
8. Begin accepting requests.

Shutdown sequence (on SIGTERM/SIGINT):
1. Stop accepting new connections.
2. Signal in-flight requests to complete or abort within the grace period.
3. Stop background workers, allowing in-progress jobs to finish or checkpoint.
4. Flush logs.
5. Close the database (which flushes the WAL).
6. Exit with status 0.

## 7. CLI Client — `tssp`

### 7.1 Global behavior

- The CLI MUST work with no configuration on first run, prompting the user to run `tssp init` or to set `--host` explicitly.
- All commands MUST support `--help` and produce useful, example-rich help text.
- All commands MUST support `--json` to emit machine-readable output, intended for scripting. JSON output is a stable contract.
- All commands MUST support `--quiet` (suppress non-error output) and `--verbose` (emit debug-level logs to stderr).
- All commands MUST support `--no-color` and respect `NO_COLOR`.
- All commands MUST exit with a documented exit code (see 7.4).
- All long-running commands MUST handle Ctrl+C gracefully: cancel in-flight uploads, clean up partial state on the server, print a clear cancellation message.
- All commands that talk to the server MUST set a connect timeout (default 5 s) and a read timeout (default 60 s, configurable), and MUST retry transient failures with exponential backoff (configurable, default 3 retries).

### 7.2 Configuration

The CLI configuration includes at minimum:
- `host` — the daemon's hostname or IP.
- `port` — the daemon's port (default 8421, chosen to avoid common conflicts).
- `scheme` — `http` or `https` (default `http` for local networks).
- `discovery` — boolean, enable mDNS to find the daemon by `tsspd.local` (default true).
- `timeout` — read timeout in seconds.
- `parallel` — default parallel upload count for batch operations.
- `color` — `auto`, `always`, `never`.
- `output` — default output format: `pretty` or `json`.

Configuration precedence (highest wins): CLI flag → environment variable (`TSSP_<KEY>`) → config file → built-in default.

### 7.3 Output formatting

Default output is human-readable: bordered tables for lists, key-value blocks for single entities, colored success/error markers, file sizes in IEC binary units (KiB, MiB, GiB) by default with a `--si` flag for SI units, durations in a human-friendly format ("2m 15s"), timestamps in local time with an ISO-8601 option.

JSON output is a stable schema, versioned alongside the API. Every JSON response includes a `schema_version` field.

When stdout is not a TTY, default output drops colors and bordered tables, switching to tab-separated values for lists. This makes pipelines work without flags.

### 7.4 Exit codes

- `0` — success.
- `1` — generic error (uncategorized).
- `2` — usage error (bad flags, missing arguments). Matches POSIX convention.
- `3` — configuration error.
- `4` — network error (cannot reach daemon).
- `5` — server error (daemon returned 5xx).
- `6` — not found (the requested file or resource does not exist).
- `7` — conflict (duplicate, already exists, version mismatch).
- `8` — permission denied (local filesystem).
- `9` — cancelled by user (Ctrl+C).
- `10` — partial success (batch operation with some failures; details in output).

### 7.5 Command Reference

This is the complete, locked CLI surface. Each command is specified by: synopsis, behavior, flags, output, and edge cases.

---

#### `tssp <file>`

**Synopsis:** Upload a single file (default action, no subcommand required).

**Behavior:** Reads the file from the local filesystem, streams it to the daemon's upload endpoint, prints a confirmation with the assigned id, the deduplication status, the upload duration, and the achieved throughput.

**Flags:**
- `-tag <name>` — attach a tag (repeatable).
- `--pin` — pin the file immediately on upload.
- `--rename <new-name>` — store under a different filename than the local one.
- `--parallel <n>` — meaningful only when combined with `-r` or `-a`.

**Output:** One line per file with a success marker, the filename, the assigned id, the size, and a deduplication indicator when applicable.

**Edge cases:** see §10.

---

#### `tssp -r <folder>`

**Synopsis:** Recursively upload a folder.

**Behavior:** Walks the folder, builds a manifest (counts, total size), prompts for confirmation if the total exceeds a configurable threshold (default 1 GB), then uploads concurrently with the configured parallelism. Progress is shown as an aggregate progress bar with current-file detail underneath. Symlinks are NOT followed by default (configurable). Hidden files (dotfiles) are skipped by default (configurable via `--hidden`). The folder structure is preserved as logical path metadata in the index (it does not dictate on-disk layout on the server).

**Flags:** all the flags of `tssp <file>` plus:
- `--follow-symlinks` — opt into following symlinks.
- `--hidden` — include hidden files.
- `--exclude <pattern>` — gitignore-style pattern, repeatable.
- `--dry-run` — show what would be uploaded without uploading.

**Output:** Aggregate progress bar, per-file completion lines, final summary (uploaded / deduplicated / failed / skipped, with totals).

---

#### `tssp -a`

**Synopsis:** Upload all non-hidden files in the current directory (non-recursive).

**Behavior:** Equivalent to `tssp -r .` but without descending into subdirectories.

**Flags:** same as `tssp -r`.

---

#### `tssp send <file>`

**Synopsis:** Upload a file and immediately push it to the user's primary phone session.

**Behavior:** Uploads the file (deduplicating if already present), creates a send session, and triggers a notification to the active web app session on the registered phone so the file auto-downloads. If no active web session is detected, falls back to QR mode automatically and prints an informational notice.

**Flags:**
- `-qr` — force QR mode regardless of active session. Renders a QR code in the terminal and prints the equivalent URL underneath for terminals that can't render the QR.
- `-tag <name>` — attach a tag (repeatable).

**Output:** Confirmation of upload, then either "Sent to phone" with the session id, or a QR rendering and URL.

---

#### `tssp receive`

**Synopsis:** Create a receive session and wait for the phone to upload a file.

**Behavior:** Creates a receive session, renders a QR code in the terminal with the upload URL, polls the daemon for session completion, prints the uploaded file's metadata when it arrives, and exits. Honors a configurable maximum wait (default: the session TTL). Ctrl+C cancels the session on the server.

**Flags:**
- `--save <path>` — also download the uploaded file to the given local path.
- `--timeout <duration>` — override the maximum wait.

**Output:** QR rendering, URL, "Waiting..." indicator, then the file record on completion.

---

#### `tssp paste`

**Synopsis:** Upload the contents of the system clipboard.

**Behavior:** Inspects the clipboard. If it contains an image, uploads it as a PNG with a generated filename including timestamp. If it contains text, uploads as a `.txt` file (UTF-8). If it contains a file path or URI list (Linux/macOS file managers put these on the clipboard when you copy files in a file browser), uploads each referenced file. If the clipboard is empty or contains an unsupported type, exits with a clear error.

**Flags:**
- `-tag <name>` — attach a tag.
- `--as <filename>` — override the generated filename.

---

#### `tssp copy <id>`

**Synopsis:** Copy a file's download URL to the system clipboard.

**Behavior:** Looks up the file by id, constructs a download URL using the configured host/port, places it on the clipboard, prints the URL to stdout for confirmation.

**Flags:**
- `--share` — instead of the direct download URL, create a short-lived shareable session URL and copy that.

---

#### `tssp pull <id|name>`

**Synopsis:** Download a file from the daemon.

**Behavior:** Resolves the argument: if it matches an id exactly, uses it; otherwise treats it as a filename and disambiguates (most recent match wins, with a warning if multiple matches exist; `--all` downloads all matches). Streams to the current directory, preserving the original filename. Refuses to overwrite by default; `--overwrite` permits it; `--output <path>` directs to a specific location. Supports HTTP Range to resume an interrupted download.

**Flags:**
- `--output <path>` — destination path.
- `--overwrite` — replace existing local file.
- `--all` — download all matches when the argument is ambiguous.

---

#### `tssp list`

**Synopsis:** List files.

**Behavior:** Fetches a page of files from the daemon, defaulting to recent-first ordering. Renders as a table (id, name, size, uploaded-at, tags). Supports filtering and ordering flags. Supports cursor-based pagination via `--page <cursor>` printed at the bottom of each page.

**Flags:**
- `--tag <name>` — filter by tag (repeatable, AND semantics).
- `--type <mime-prefix>` — e.g., `image`, `video`, `application/pdf`.
- `--since <duration>` — e.g., `1d`, `2h`, or ISO timestamp.
- `--limit <n>` — page size, default 50, max 500.
- `--sort <field>` — `uploaded`, `name`, `size`, `-uploaded` (descending), etc.
- `--pinned` — pinned files only.

---

#### `tssp last [n]`

**Synopsis:** Show the most recent `n` uploads (default 10).

**Behavior:** Equivalent to `tssp list --sort -uploaded --limit <n>` but with a more compact default output.

**Synopsis:** Show files uploaded today (in the user's local timezone).

**Flags:**

**Behavior:** Adds tags to the file. Idempotent. Validates tag names.

**Synopsis:** Remove a tag from a file.

**Behavior:** Removes the tag. Idempotent.

**Synopsis:** Pin a file to favorites.

**Flags:**
- `--position <n>` — insert at a specific position in the pin order.

**Synopsis:** Delete a file from the daemon.

**Behavior:** Prompts for confirmation unless `--yes` is passed or stdin is not a TTY. Idempotent.

**Flags:**

**Synopsis:** Show full details for a file.

**Behavior:** Renders a key-value block with id, name, size, hash, MIME type, uploaded-at, source device (if recorded), tags, pin status, deduplication info (other ids sharing the same content if any), and a direct download URL.

## 10. Edge Cases — Comprehensive

This section is the heart of the specification. The implementer MUST handle every case below explicitly, with a test that demonstrates correct behavior. "It probably won't happen" is not a valid response.

### 10.1 Network and addressing

- **Daemon IP changes (DHCP).** When the Pi reboots and gets a new IP, the CLI's saved `host` becomes wrong. The CLI MUST support hostname-based addressing and MUST attempt mDNS discovery of `tsspd.local` as a fallback when the configured host fails to connect. `tssp init` MUST prefer hostnames over raw IPs.
- **IPv4 and IPv6.** The daemon MUST bind to both families when configured to bind to `[::]:port`. The CLI MUST resolve hostnames and try addresses in order, with a per-address timeout.
- **mDNS unavailable.** Some networks block mDNS. The CLI MUST fall back cleanly and produce a clear error pointing at how to set `host` manually.
- **DNS resolution failure.** Distinguish between "name does not resolve" and "host unreachable" in error messages.
- **Captive portal / network requires login.** Connection attempts that return HTML or unexpected content types MUST produce a clear "looks like a captive portal" error rather than a parse failure.
- **Wi-Fi roaming mid-upload.** TCP connection drops. The CLI MUST retry from the last acknowledged byte using HTTP Range on resumable downloads, and MUST retry the upload from scratch (or, future work, with a resumable upload protocol) with backoff.
- **Slow networks.** Default timeouts MUST be generous enough for a 100 MB upload over a weak link. Progress MUST keep showing throughput so the user can tell the upload is alive.
- **Server behind reverse proxy.** Path stripping and `X-Forwarded-*` headers MUST be respected so the daemon generates correct absolute URLs.
- **Port already in use.** On startup, the daemon MUST fail with a clear message indicating the port and suggesting how to change it.
- **Firewall blocking.** Connection-refused versus connection-timeout MUST be distinguished in CLI error messages.
- **Clock skew.** Client and server clocks may differ. Timestamps in responses are server-side UTC; the CLI converts for display. The daemon MUST NOT reject requests based on client clock.

### 10.2 Hardware and OS constraints

- **Thermal throttling.** The daemon MUST NOT spawn unbounded worker tasks. Thumbnail generation, hashing, and image decoding MUST be governed by a configurable semaphore.
- **SD card wear and slow writes.** The storage subsystem MUST batch metadata writes where safe, MUST use SQLite WAL mode (which reduces write amplification), MUST NOT fsync per file when a batch is in progress, and MUST fsync at safe checkpoints.
- **SD card corruption.** On startup, the daemon MUST run an integrity check on the database (`PRAGMA integrity_check`) and report failure clearly. A periodic background scan MUST verify that index records point to existing blobs; missing blobs are logged as corruption and exposed via the status endpoint.
- **Storage full.** Uploads MUST fail with HTTP 507 (Insufficient Storage) and a clear message. The daemon MUST refuse to write when free space falls below a configurable safety margin (default 500 MB or 1% of total, whichever is greater).
- **Permission denied on data directory.** Detected at startup; daemon refuses to start with a clear remediation hint.
- **Read-only filesystem.** Same as above.
- **Out of memory.** The streaming design prevents the daemon from holding entire files. Any unbounded buffering (e.g., loading an entire response into a String) is a bug.
- **Process killed (OOM killer, power loss, SIGKILL).** No data corruption. On next startup the daemon detects and cleans up: temp upload files older than 1 hour, sessions past their TTL, index records with no corresponding blob (marked as corrupt, not auto-deleted).

### 10.3 Filesystem semantics

- **Filenames with unusual characters.** Spaces, quotes, newlines, control characters, slashes, backslashes, NUL. Stored filename is sanitized for filesystem-safe representation; the original is preserved verbatim in the index and shown to users.
- **Unicode filenames.** Full UTF-8 support. Normalization form NFC for storage to avoid macOS NFD vs Linux NFC mismatches.
- **Very long filenames.** Some filesystems limit individual components to 255 bytes. The daemon MUST handle by truncating the on-disk name while preserving the original in the index.
- **Case sensitivity differences.** Index lookups by name are case-insensitive; the on-disk layout is case-preserving but treats names as opaque.
- **Symlinks in uploads.** The CLI does not follow symlinks by default (see `-r` flag).
- **Special files (sockets, FIFOs, devices) in uploads.** The CLI MUST skip these with a warning, never block trying to read them.
- **Files that change during upload.** The hash computed during streaming reflects what was uploaded. The CLI MUST NOT re-read the file after upload.
- **Empty files.** Allowed. Hash of empty content is well-defined. Stored and indexed normally.
- **Zero-permission files.** The CLI fails the individual file with a clear permission-denied message and continues the batch.

### 10.4 Concurrency

- **Two uploads of identical content arriving simultaneously.** The deduplication check is a transaction. Exactly one stored blob, two index records, both succeed.
- **Upload while same id is being deleted.** The delete acquires a lock on the index record; the upload (which targets a hash, not an id) is unaffected. A download of the deleting id may either complete or fail with 410 depending on timing — both are correct.
- **Tag added and removed simultaneously.** Last-write-wins on the index, both operations return success.
- **Batch upload partial failure.** The overall operation returns 200 with per-item status. Successful items are committed; failed items leave no partial state.
- **Database locked errors.** SQLite under WAL mode tolerates concurrent readers and one writer. Writers MUST queue with bounded waits. A writer that cannot acquire the lock within the timeout retries with backoff; if all retries fail, returns 503 Service Unavailable to the client.
- **Background workers vs request path.** Thumbnail generation, session reaping, and integrity scans MUST yield to request handlers; they MUST be rate-limited or scheduled to avoid CPU contention.

### 10.5 CLI input handling

- **Globs (`tssp *.jpg`).** Globs are expanded by the shell; the CLI MUST handle being passed many arguments. On Windows, where globs are not shell-expanded by default, the CLI MUST expand them itself for arguments that look like glob patterns.
- **No matching files.** Clear "no files matched" error, exit code 6.
- **Argument is a directory but `-r` was not passed.** Clear error suggesting `-r`, exit code 2.
- **Argument is a directory and `-r` was passed but the directory is empty.** Clear "directory is empty" notice, exit code 0 with no upload.
- **Mixed files and directories.** With `-r`, both are processed: files uploaded directly, directories descended.
- **Relative paths and tilde expansion.** Handled by the shell on Unix; the CLI MUST handle `~` expansion on Windows manually.
- **Ctrl+C during upload.** Cancel signal sent to the server (if upload was registered with a temp id), local progress bar cleared, partial uploads cleaned up server-side, exit code 9.
- **Ctrl+C during interactive prompt.** Cancellation, no config changes saved, exit code 9.
- **Stdin is a pipe.** No interactive prompts; commands that require confirmation (`tssp remove`) require `--yes`.
- **Stdout is a pipe.** Progress goes to stderr; data output is plain (no colors, no borders). Tables become TSV.
- **Terminal does not support Unicode.** QR rendering falls back to ASCII characters; tables use ASCII box-drawing.
- **Very narrow terminal.** Tables truncate columns with ellipsis, with the most important columns prioritized.

### 10.6 QR sessions

- **QR code expires before scan.** Phone hits the URL after expiration → server returns a friendly page explaining the link expired. CLI's `receive` waiter times out and prints a clear message.
- **Multiple scans of the same QR.** Sessions are single-use. The second scan returns "this link has already been used."
- **CLI killed while waiting for receive.** The receive session is cancelled on the server (best-effort via a cancellation request) and reaped within 60 seconds even if cancellation fails.
- **Phone uploads a file larger than the server's max upload size.** Clear error on the phone's page; session remains valid until TTL.

### 10.7 Clipboard

- **Clipboard contains an image.** Saved as PNG.
- **Clipboard contains text.** Saved as `.txt`.
- **Clipboard contains a file URI list.** Each referenced file is uploaded.
- **Clipboard contains multiple types (e.g., HTML with image fallback).** Prefer image > file list > text.
- **Clipboard is empty.** Clear error, exit code 2.
- **Headless system (no X11/Wayland).** Clear "no display server available" error, exit code 3.
- **Clipboard contents change during read.** Single atomic snapshot read; subsequent changes do not affect the upload.

### 10.8 Data integrity

- **Hash mismatch on download.** The CLI MAY (configurable) re-hash downloaded content and verify against the server-reported hash; mismatch is an error.
- **Truncated upload (client disconnected mid-stream).** Server detects via Content-Length mismatch, discards the temp file, returns 400, leaves no orphan in the index.
- **Bit rot on disk.** Detected by the periodic integrity scan; affected files are marked corrupt in the index and surface in `tssp status`.

### 10.9 Versioning and upgrades

- **Daemon upgrade with database schema change.** Migrations run at startup; if migration fails, daemon refuses to start and the previous version's database is left untouched (migrations operate on a copy or in a transaction).
- **Daemon and CLI version mismatch.** The CLI sends its version in a header; the daemon includes its version in every response. On significant mismatch, the CLI prints a one-time warning.
- **Old CLI talking to new daemon.** Backward compatibility maintained within a major version.
- **New CLI talking to old daemon.** The CLI detects unsupported endpoints via 404 and prints a clear "your daemon is too old, please upgrade to vX.Y" error.

### 10.10 Localization and formatting

- **Number formatting.** No locale-dependent separators by default (avoid `1,234` vs `1.234` confusion). User can opt in via config.
- **Date formatting.** ISO-8601 by default, with a human-friendly relative form ("3 minutes ago") in interactive output.
- **RTL languages in filenames.** Stored and displayed correctly without breaking layout.

## 12. Security Model (v1)

The trust model for v1 is **local network only**. The daemon listens on the LAN, there is no authentication, and the user is expected to trust everyone on the network. This MUST be:

- Documented prominently in the README.
- Reflected in the default bind address (the daemon SHOULD default to binding only to private IPv4 ranges and link-local IPv6, refusing to bind to public addresses unless explicitly configured).
- Reinforced by `tssp init` warning the user if it detects the daemon is reachable from a public interface.

QR session tokens are 128 bits of entropy, single-use, and short-lived. This is sufficient to prevent guessing attacks within the session TTL.

Path traversal in filenames is prevented at every boundary: filenames are sanitized when forming on-disk paths; the on-disk layout uses opaque identifiers, not user-supplied names; downloads stream from validated handles, not user-controlled paths.

The HTTP server MUST set: `X-Content-Type-Options: nosniff` on all responses; `Content-Security-Policy` on web app responses; appropriate CORS headers configured via config.

Authentication, multi-user support, TLS, encryption at rest — explicitly out of scope for v1. The architecture MUST permit them as future additions (§5.3).
